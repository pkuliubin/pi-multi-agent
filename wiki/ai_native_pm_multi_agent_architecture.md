# AI Native PM 协作流程：多 Agent 架构选型笔记

## 1. 背景：传统 PM - 研发协作流程的问题

传统的产品经理（PM）与研发协作流程通常是：

1. PM 设计 PRD
2. PM 与研发开会解释产品需求
3. 双方在会上进行大量讨论
4. 研发会后再设计技术方案并实施

这个过程中经常出现以下问题：

- 产品方案一开始想不清楚
- 方案本身不合理，或者隐藏了关键假设
- 方案实现难度过高，但在需求前期没有被及时识别
- 时间紧迫时，研发临近实现阶段才发现 scope 与成本不匹配
- PM 和研发使用不同的话语体系，沟通摩擦大
- 中间讨论没有被很好沉淀，后续容易重复沟通
- 需求目标、实现方式、边界条件在流转中逐渐失真

## 2. 一个更 AI Native 的工作流设想

相比传统流程，一个更 AI Native 的产品协作工作流可以是：

### 阶段 1：PM 先和多个专家 Agent 讨论需求

PM 不直接先找研发开会，而是先与多个不同角色的 Agent 进行多轮讨论，例如：

- 资深产品 Agent
- 全栈研发 Agent
- UI / UE Agent
- 数据分析 Agent

这些 Agent 会从各自视角对新的产品想法进行质疑、建议和补充：

- 产品 Agent：挑战需求定义、目标清晰度、用户价值、优先级
- 研发 Agent：结合现有代码库判断是否可实现、实现成本、技术约束、替代方案
- UI / UE Agent：判断交互复杂度、用户路径、认知负担、边界体验
- 数据分析 Agent：结合历史数据分析需求是否打到真实痛点、是否能达成目标、指标如何设计

经过多轮沟通后，PM 会形成一套较完整的产物：

- 最终版 PRD 草稿
- 明确的业务目标与 success metrics
- 已识别的关键假设
- 潜在技术方案
- open questions 列表
- 讨论与决策历史

### 阶段 2：研发先与 Agent 对接理解需求

需求确定后，不是 PM 先直接和研发对接，而是把上述 artifacts 交给研发。

研发先和 Agent 对话，理解需求目标、设计意图、约束和上下文；如果仍有关键疑问，Agent 无法回答，再与 PM 当面沟通。

### 阶段 3：研发确认无误后开始设计与开发

研发确认需求没有关键歧义后，再进入技术方案设计和实施。

## 3. 这个流程试图解决什么问题

这个 AI Native 工作流希望提前把许多传统沟通成本前置并结构化，解决例如：

- 产品在需求阶段没有真正想清楚
- 技术实现难度过高但发现太晚
- 方案无法兼顾产品目标、用户体验和实现成本
- PM 和研发语言体系不一致导致误解
- 历史讨论没有沉淀，后续反复沟通
- 目标、需求、方案、约束无法形成统一上下文

# 4. 聚焦第一个环节：PM 与多个 Agent 一起完善需求

如果只聚焦第一个环节，也就是“PM 和一堆 Agent 一起完善产品需求”，最合适的不是单一模式，而是一个**混合型 multi-agent 架构**：

> **以 Orchestrator–Workers 为主骨架，叠加 Shared State，并在收尾阶段加入 Generate–Verify。**

也可以写成：

```text
Orchestrator–Workers + Shared State + Generate–Verify
```

## 5. 为什么这是最合适的架构

### 5.1 主骨架：Orchestrator–Workers

这个阶段的核心目标不是让很多 Agent 同时自由发言，而是：

- 围绕一个产品想法持续澄清
- 把不同视角的质疑和建议汇总起来
- 推动讨论一轮轮收敛
- 最终产出一份可进入下游流程的 PRD / 需求包

因此需要一个中心 Agent 作为 orchestrator，例如：

- PM facilitator agent
- product orchestration agent
- requirements synthesis agent

它负责：

- 理解当前产品想法
- 拆分本轮需要讨论的问题
- 调用各个专家 Agent 获取意见
- 汇总冲突点和共识
- 驱动下一轮追问
- 维护当前版本的需求状态

之所以不是让 PM 直接分别和多个 Agent 自由对话，是因为那样很容易：

- 信息分散
- 结论冲突
- PM 认知负担过大
- 无法自然收敛

更合理的方式是：

- **PM 主要只与 orchestrator 对话**
- orchestrator 再去调度产品、研发、UI/UE、数据等专家 Agent
- orchestrator 把结构化结论返还给 PM

### 5.2 状态层：Shared State

如果只有 orchestrator 和 workers，但没有共享状态，讨论依然容易反复、丢失上下文。

因此需要一套共享 artifacts，作为所有 Agent 的共同工作面。这些共享状态不应只是长聊天记录，而应是结构化文档，例如：

- PRD draft
- 用户问题定义
- 目标与 north star / success metrics
- 关键假设列表
- 非目标（out of scope）
- 约束条件
- 技术风险与成本评估
- 交互草案
- open questions
- 决策记录（decision log）

每轮讨论的目标，不只是“多输出一点意见”，而是**更新这套共享产物**。

这样做的好处是：

- 后续研发接手时有明确上下文
- 中间结论不会只留在聊天里
- 冲突可以被显式记录
- 后续可以只讨论 unresolved issues，而不是从头重来

### 5.3 收口层：Generate–Verify

即使经过多轮讨论，也不代表需求已经“足够清楚、足够可落地”。

所以在 PRD 初稿或需求包形成后，建议增加一层 verifier / skeptic agent，对结果进行 readiness check，例如检查：

- 目标是否清晰
- 指标是否可验证
- 用户问题是否真实存在
- 方案是否与目标一致
- 技术实现是否可行
- scope 是否与时间窗口匹配
- 是否仍有未关闭的关键歧义

这一步是把“讨论过程”转换成“可交付结果”的关键。

# 6. 为什么不是其他单一架构

## 6.1 为什么不是纯 Agent Teams

Agent Teams 更适合：

- 大量长期并行工作
- 各个子任务相对独立
- 每个 Agent 长期负责一个模块或工作域

而当前这个阶段并不是长期执行任务，而是围绕**同一个需求对象进行反复对齐和收敛**。它更需要一个中心收敛机制，而不是多个长期自治 worker。

## 6.2 为什么不是纯 Shared State

如果只有共享状态，没有 orchestrator：

- 不同专家 Agent 可能各写各的
- 讨论方向容易发散
- 冲突无人收敛
- 容易进入循环讨论，不知道什么时候结束

所以 Shared State 很重要，但它更适合作为协作底座，而不是唯一架构。

## 6.3 为什么不是 Message Bus

Message Bus 更适合事件驱动系统，例如：

- 有新告警就触发某类 Agent
- 有新工单就由不同 Agent 接力处理
- 新事件不断进入系统，由订阅者响应

而 PM 完善产品需求这个环节，不是典型事件驱动问题，而是**围绕单一对象进行多轮 deliberation 和收敛**。因此消息总线并不是第一选择。

## 6.4 为什么不是只用 Generate–Verify

Generate–Verify 很适合提高输出质量，但它主要解决的是：

- 生成完后如何检查
- 是否符合标准
- 如何迭代修正

它不擅长组织多专业视角的持续讨论，也不能自然承担“主持人”角色。因此它适合作为最后的验收层，而不是完整主架构。

# 7. 推荐的第一环节工作流

## Step 1：Orchestrator 定义当前问题

orchestrator 首先将 PM 当前的产品想法转化为结构化问题，例如：

- 要解决的核心用户问题是什么
- 目标是什么
- 为什么现在要做
- 当前方案有哪些关键假设
- 还缺哪些信息

## Step 2：专家 Agent 分角色给出审查意见

不同专家 Agent 分别从自己的角度审查和补充：

- 产品 Agent：需求价值、优先级、范围合理性
- 研发 Agent：代码库适配性、实现复杂度、替代方案、风险
- UI / UE Agent：用户路径、交互负担、边界场景
- 数据 Agent：真实痛点、数据支撑、指标设计、实验可能性

## Step 3：Orchestrator 汇总冲突并推动下一轮

orchestrator 不只是把意见拼起来，而是要：

- 识别共识和冲突
- 把问题压缩成下一轮最值得追问的 unresolved issues
- 驱动 PM 澄清或取舍
- 更新共享产物

## Step 4：Verifier 做 PRD readiness check

当共享产物达到一定完整度后，由 verifier 对需求包做最后验证，判断：

- 现在是否已经足够交给研发继续推进
- 哪些问题还必须关闭
- 哪些只是次要不确定性，可以留到设计或开发阶段解决

# 8. 这种架构的本质

这个场景本质上不是“让多个 Agent 并行工作”这么简单，而更像：

- 多专家参与的需求评审
- 有主持人的澄清与收敛过程
- 共享决策文档的持续演化
- 最终通过验收门槛的需求交付

因此，最重要的并不是“Agent 数量”，而是以下三个机制是否存在：

1. **Decision log**：重要结论是否被记录
2. **Assumption log**：关键假设是否被显式化
3. **Readiness check**：是否存在明确的交付判定

如果没有这三层，即使 Agent 很聪明，讨论仍然容易变成高质量但不可沉淀的聊天。

---

# 9. 一句话结论

对于“PM 与多个 Agent 一起完善产品需求”这个第一环节，最合适的 multi-agent 架构是：

> **Orchestrator–Workers 负责讨论收敛，Shared State 负责沉淀上下文，Generate–Verify 负责最终验收。**

这是一个比单一模式更贴近实际业务流程的混合架构。

# 10. Claude Blog《Multi-agent coordination patterns》内容摘要

文章链接：<https://claude.com/blog/multi-agent-coordination-patterns>

## 10.1 文章主旨

这篇文章的核心观点是：多智能体系统不应该先追求更复杂，而应该先选**足够简单且能工作的协同方式**，然后再根据实际瓶颈逐步演进。

作者强调，设计 multi-agent 系统时应重点关注两件事：

1. 上下文如何切分
2. 信息如何流动

换句话说，多 Agent 架构设计的重点不是“用了多少 Agent”，而是：

- 每个 Agent 需要什么上下文
- 中间结果如何传递
- 冲突如何处理
- 系统如何收敛

## 10.2 五种协同模式

### 1) Generate–Verify

**适合解决：** 输出质量要求高，且验收标准可以清晰定义的任务。

**工作方式：**

- 一个 Agent 先生成结果
- 另一个 Agent 按标准检查
- 如果不合格，反馈并重试
- 直到通过或达到最大轮次

**典型场景：**

- 客服回复
- 代码生成 + 测试
- 合规检查
- 事实核查
- 按 rubric 评分的内容生成

**权衡：**

- 验证标准不清时，检查会流于形式
- 如果“判断对错”和“生成内容”一样难，验证方未必可靠
- 可能反复修改却不收敛，因此应设置最大迭代次数和兜底方案

### 2) Orchestrator–Workers

**适合解决：** 任务可以明确拆解，子任务边界清楚、依赖较少的场景。

**工作方式：**

- 主 Agent 负责规划、分派和汇总
- 多个子 Agent 分别处理专项任务
- 主 Agent 最终整合结果

**典型场景：**

- 自动代码审查
- 代码库大范围搜索
- 多来源信息收集与汇总

**权衡：**

- 主 Agent 可能成为信息瓶颈
- 子 Agent 之间不能直接共享细节时，信息容易丢失
- 如果并行度不足，收益不一定高于复杂度

### 3) Agent Teams

**适合解决：** 可并行、彼此独立、且需要持续多步推进的长任务。

**工作方式：**

- 协调者启动多个长期存活的 worker agents
- 它们从共享任务池领取任务
- 每个 worker 在自己的工作域持续推进，并保留长期上下文

**典型场景：**

- 大型代码库迁移
- 多服务并行改造

**权衡：**

- 前提是任务之间足够独立
- 中间发现不易共享
- 可能做出冲突决策
- 如果共享代码库或数据库，可能发生资源冲突

### 4) Message Bus

**适合解决：** 事件驱动流程，并且参与的 Agent 种类会持续增加的系统。

**工作方式：**

- Agent 通过发布 / 订阅消息进行交互
- 路由器把消息分发给相关订阅方

**典型场景：**

- 安全运营自动化
- 告警分类和后续处理

**权衡：**

- 灵活，但调试和追踪更困难
- 路由质量非常关键
- 如果用大模型做语义路由，会引入新的失误模式

### 5) Shared State

**适合解决：** 协作式任务，多个 Agent 需要实时利用彼此发现的场景。

**工作方式：**

- 多个 Agent 直接读写同一持久化存储
- 不通过中心编排者
- 通过终止条件、时间限制或收敛标准结束工作

**典型场景：**

- 研究综合
- 共享知识积累

**权衡：**

- 容易重复劳动或相互矛盾
- 更危险的是反应式循环：一个写入触发另一个，再反过来触发前者
- 因此终止条件必须前置设计

## 10.3 文章强调的选型逻辑

作者给出的关键判断维度包括：

- 任务拆分应按**所需上下文**来做，而不只是按职能名称拆分
- Orchestrator–Workers vs Agent Teams：看子任务是否需要跨多轮保留上下文
- Orchestrator–Workers vs Message Bus：看流程是预先规划还是由事件动态决定
- Agent Teams vs Shared State：看成员之间是否需要实时共享中间发现
- Message Bus vs Shared State：看系统处理的是离散事件还是持续积累的公共知识

## 10.4 文章的总体建议

文章建议：

- 先从最简单可行的模式开始，不要一开始就堆复杂度
- 大多数场景先从 Orchestrator–Workers 起步
- 生产环境中经常会采用混合架构，而不是五选一

## 10.5 文章给人的关键启发

这篇文章最有价值的地方在于，它把 multi-agent 问题重新落回到软件架构问题：

- 边界怎么划
- 信息怎么流
- 冲突怎么解
- 状态怎么沉淀
- 系统怎么停

因此，多 Agent 系统更像是在设计一个协作式工作流 / 分布式决策系统，而不只是“让多个模型一起回答问题”。

# 11. 最后结论

如果把你的场景看成一个 AI Native 的产品需求打磨系统，那么第一阶段最合理的设计不是“让一群 Agent 都能发言”，而是：

- 用 **orchestrator** 控制讨论节奏和收敛
- 用 **shared artifacts** 承载上下文、决策和未决问题
- 用 **verifier** 判断需求是否已达到可交付状态

这三层结合起来，才更有可能真正降低 PM 与研发之间的沟通摩擦，并把中间推理过程变成可复用资产。
