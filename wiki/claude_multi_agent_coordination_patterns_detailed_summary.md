# Claude 博客《Multi-agent coordination patterns》详细解读

原文链接：<https://claude.com/blog/multi-agent-coordination-patterns>

## 1. 这篇文章在讲什么

这篇文章讨论的不是“多智能体是不是更强”，而是：

> 当一个任务需要多个 Agent 协作时，应该如何设计它们之间的关系？

作者的核心观点非常明确：

- 多 Agent 系统并不天然优于单 Agent
- 只有当任务确实能从拆分中获益时，多 Agent 才值得引入
- 架构设计时，最重要的不是“有多少个 Agent”，而是：
  - **上下文如何切分**
  - **信息如何流动**
  - **冲突如何处理**
  - **系统如何收敛**

换句话说，这篇文章本质上是在讨论 multi-agent 系统的**协同拓扑结构**。

作者没有鼓励大家一上来就做复杂系统，反而强调：

- 从最简单可行的模式开始
- 只有当简单模式暴露瓶颈时，再引入更复杂的结构
- 很多生产系统最终会是混合架构，而不是单一模式

---

# 2. 五种 multi-agent 协调模式总览

文章总结了五种常见的协同模式：

1. **Generate–Verify**
2. **Orchestrator–Workers**
3. **Agent Teams**
4. **Message Bus**
5. **Shared State**

它们并不是按“先进程度”排序，而是分别适配不同的问题结构。

如果用一句话概括：

- **Generate–Verify**：先做，再审，再改
- **Orchestrator–Workers**：中心拆解，分工完成，再汇总
- **Agent Teams**：多位长期成员并行推进独立工作
- **Message Bus**：由事件驱动，不同 Agent 按订阅响应
- **Shared State**：多个 Agent 围绕同一共享知识面协作

下面逐一展开。

---

# 3. 模式一：Generate–Verify

## 3.1 模式定义

Generate–Verify 是最直观的一种模式：

- 一个 Agent 负责生成结果
- 另一个 Agent 负责验证结果是否符合标准
- 若不符合，则生成 Agent 根据反馈重新修改
- 如此迭代，直到通过或达到轮次上限

它可以理解为一种“生产者 + 质检者”的结构。

## 3.2 它解决什么问题

这种模式特别适合：

- 输出质量要求高
- 质量标准相对明确
- 可以通过规则、测试、检查表、rubric、政策约束等方式判断结果好坏

它不强调任务拆分，而强调**结果把关**。

## 3.3 典型场景

文章举出的场景包括：

- 客服邮件回复生成后再审校
- 代码生成后跑测试或静态检查
- 事实核查
- 内容是否符合某个评分标准
- 合规审查

把它推广开来，还适用于：

- SQL 生成 + 语义校验
- 报告生成 + 结构完整性检查
- PRD 生成 + 模板/逻辑一致性验证
- 文本改写 + 品牌语气检查

## 3.4 优点

### 1）实现简单

这是五种模式里最容易理解和落地的之一。你不需要复杂的分工结构，只需要定义：

- 谁生成
- 谁验证
- 验证失败后怎么反馈
- 最多改几轮

### 2）非常适合提升质量

很多任务第一次输出就能“基本可用”，但离“稳定高质量”还差一点。Generate–Verify 正好能补这个 gap。

### 3）对已有流程侵入小

它可以加在很多现有单 Agent 流程的末尾，不需要彻底重做架构。

## 3.5 局限与风险

### 1）验证标准不清时会失效

如果连人都说不清“什么叫好”，那 verifier 很容易变成泛泛而谈：

- 再清楚一点
- 再详细一点
- 再优化一下

这种反馈不会带来真正收敛。

### 2）“验证”和“生成”一样难时，验证未必可信

如果问题本身极复杂，判断对错和生成内容一样困难，那么 verifier 未必真的更可靠。

例如：

- 高难度开放性研究结论
- 高不确定性的战略判断
- 缺少外部可验证标准的创作类任务

### 3）容易来回反复，不收敛

如果生成者和验证者都没有强约束，系统会陷入反复修改：

- verifier 提一点模糊意见
- generator 修一点
- verifier 再提一点
- 循环继续

## 3.6 设计时要注意什么

文章特别强调：

### 设置最大迭代次数

不能让它无限循环。要有明确上限，例如：

- 最多 3 轮
- 最多 5 轮

### 设置兜底策略

如果始终无法通过，应明确后续动作，例如：

- 返回当前最佳版本
- 输出未通过原因
- 升级为人工审核
- 降级采用保守方案

### 尽量结构化验证标准

越能把“好坏”转成可检查的规则、清单、测试，效果越好。

## 3.7 适合什么时候用

当你面对的是：

> “一个结果已经能产出，但我担心它不够稳、不够准、不够合规。”

这时优先考虑 Generate–Verify。

# 4. 模式二：Orchestrator–Workers

## 4.1 模式定义

Orchestrator–Workers 是最经典的“主控 + 分工”模式：

- 一个主 Agent（orchestrator）理解整体任务
- 它把任务拆成多个子任务
- 分派给多个 worker agents
- 等 worker 返回结果后，再由 orchestrator 汇总成最终输出

它可以理解为“项目经理 + 专家执行者”的结构。

## 4.2 它解决什么问题

这种模式适合：

- 总任务可以被拆分
- 子任务边界相对清楚
- 各子任务之间依赖不强
- 最终结果仍需要一个中心来整合

它解决的是：

> 单个 Agent 一次性处理整个复杂任务，容易上下文过载、关注点混乱、输出不稳定。

通过任务拆分，可以让每个 worker 聚焦局部问题。

## 4.3 典型场景

文章里提到的包括：

- 自动代码审查
- 在大型代码库中做搜索和调查
- 主 Agent 在做主线任务时，后台派出多个子 Agent 做专项调查

可以推广到：

- 多文档调研后输出综合报告
- 需求分析时同时调产品、技术、设计、数据视角
- 一个 Agent 负责需求拆分，多个 Agent 分别估算复杂度、找风险、提替代方案

## 4.4 优点

### 1）最适合做复杂任务拆解

当你有一个大任务，但它天然包含几个可定义子问题时，这个模式非常自然。

### 2）中心控制强，易于收敛

orchestrator 负责把控整体质量与方向，不容易让系统完全失控。

### 3）对用户体验友好

用户不需要同时和多个 Agent 自由交互，只需要和 orchestrator 对话。

### 4）通常是最稳妥的起点

作者实际上建议：很多团队在一开始做 multi-agent 时，可以先从这个模式起步。

## 4.5 局限与风险

### 1）orchestrator 会成为信息瓶颈

所有信息都要经过 orchestrator：

- 它要会拆分
- 它要会提炼
- 它要会汇总

如果 orchestrator 本身做得不好，整个系统都会受影响。

### 2）子 Agent 之间难以直接共享细节

如果 worker A 的发现对 worker B 很重要，但两者不能直接交流，只能通过 orchestrator 转发，那么：

- 信息容易被压缩过度
- 细节容易丢失
- 协同效率可能下降

### 3）并行收益可能没有想象中大

如果子任务拆得不够独立，或者 orchestrator 花大量时间协调，系统复杂度上去了，但收益有限。

## 4.6 设计时要注意什么

### 1）先按上下文边界拆分，不是按标签拆分

不要机械地说：

- 一个做技术
- 一个做设计
- 一个做数据

更重要的是看：

- 每个子任务到底需要哪些材料
- 是否能在有限上下文中独立完成

### 2）明确 worker 的输入输出格式

worker 产出如果太自由，orchestrator 很难有效汇总。

### 3）避免让 orchestrator 承担过度总结负担

如果每个 worker 返回大段自由文本，最终会把汇总压力重新堆回主 Agent。

## 4.7 适合什么时候用

当你面对的是：

> “这是一个复杂任务，但我能比较明确地拆成几块，并且最后仍需要一个中心来拼起来。”

这时优先考虑 Orchestrator–Workers。

# 5. 模式三：Agent Teams

## 5.1 模式定义

Agent Teams 和 Orchestrator–Workers 看起来相似，但关键差异在于：

- worker 不是一次性短任务执行者
- 而是长期存在的“团队成员”
- 它们在自己的职责范围内持续工作，并保留长期上下文

可以把它理解为一个长期合作的小团队。

## 5.2 它解决什么问题

这种模式适合：

- 工作周期长
- 任务量大
- 多个子任务能长期并行推进
- 不同 Agent 长期负责不同工作域是有价值的

它解决的是：

> 某些任务不是“拆一下、做完、汇总”就结束，而是需要多个角色持续协作、连续推进。

## 5.3 典型场景

文章举的例子是：

- 大型代码库从一个框架迁移到另一个框架
- 不同服务可以由不同 agent 长期负责

可以进一步类比到：

- 多模块系统重构
- 大型运营分析体系搭建
- 长周期内容生产团队
- 多产品线持续迭代支持

## 5.4 和 Orchestrator–Workers 的区别

这是理解这两种模式最关键的一点。

### Orchestrator–Workers

- worker 常常是“短暂调用”
- 处理完一个子任务就结束
- 更像是临时咨询或专项执行

### Agent Teams

- worker 是长期存在的
- 会持续积累自己负责领域的上下文
- 更像稳定团队成员

因此，是否需要**跨多轮保留局部上下文**，是二者最关键的分界线。

## 5.5 优点

### 1）适合长期任务

如果任务周期长，频繁重新拉起同类 worker 会损失上下文与连续性。

### 2）更容易形成领域专长

每个 team member 可以在自己的工作域里持续深化。

### 3）并行能力更强

尤其是在工作天然可分区时，效率提升可能很明显。

## 5.6 局限与风险

### 1）前提是任务要足够独立

如果大家总是在相互等待、相互依赖，那长期团队制未必比中心调度更有效。

### 2）中间发现不容易共享

每个人都在自己的上下文里工作，可能会：

- 各自优化局部
- 忽略全局一致性
- 做出互相冲突的决策

### 3）完成节奏不同，协调复杂

有些 agent 先完成，有些落后，如何处理部分完成状态是一个现实问题。

### 4）共享资源会冲突

如果多个 agent 同时改同一个代码库、数据库、文档系统，就可能发生冲突。

## 5.7 设计时要注意什么

### 1）任务分区必须明确

要清楚谁负责什么边界，否则会频繁重叠。

### 2）需要冲突处理机制

例如：

- 谁拥有最终裁决权
- 冲突如何合并
- 谁来维护统一约束

### 3）协调者要能处理部分完成状态

系统不应假设所有工作会同步完成。

## 5.8 适合什么时候用

当你面对的是：

> “这是一个长周期、多模块、可并行的工作，而且每个角色保留长期上下文会持续增值。”

这时优先考虑 Agent Teams。

# 6. 模式四：Message Bus

## 6.1 模式定义

Message Bus 模式借鉴的是经典发布 / 订阅系统：

- Agent 不一定彼此直接通信
- 它们通过消息总线发布消息或订阅事件
- 路由层根据主题、标签或语义，将消息发给相关 Agent

这是一种典型的事件驱动结构。

## 6.2 它解决什么问题

适合：

- 流程是事件驱动的
- 参与者会逐渐增多
- 不想让所有协作都绑定到一个中心 orchestrator
- 希望增强系统的扩展性和解耦性

它解决的是：

> 一个系统中不断有新事件发生，不同 Agent 应该按需接手处理，而不是每次都由中心规划整条流程。

## 6.3 典型场景

文章举例是安全运营自动化：

- 告警先进入系统
- 被分类
- 再分发给不同调查 Agent
- 后续补充信息、调查结果、响应动作继续通过总线传播

类似场景还包括：

- 监控告警分发
- 工单处理流水线
- 用户行为事件触发多类分析
- 客服事件路由

## 6.4 优点

### 1）扩展性强

新增一种 Agent，只要能订阅某类消息，就能接入流程，不需要大改主逻辑。

### 2）耦合度低

消息生产者不一定要知道最终由谁处理。

### 3）适合开放式生态

如果系统会不断接入新角色、新能力，这种模式很有优势。

## 6.5 局限与风险

### 1）链路难追踪

因为系统是分布式事件流，不像 orchestrator 那样有一条明显主线。

### 2）调试更困难

出了问题时，经常很难快速回答：

- 这个消息去了哪里
- 哪个 Agent 收到了
- 为什么没继续处理
- 哪一步做错了

### 3）路由质量决定系统上限

如果消息分发错了、漏了、歧义了，系统可能不会直接报错，但会静悄悄地做错事。

### 4）语义路由会引入新的错误模式

如果让 LLM 来决定消息应该发给谁，会更灵活，但也会引入模糊、误分、边界不清等问题。

## 6.6 设计时要注意什么

### 1）路由规则必须可观测

不能只依赖“模型觉得应该发给谁”，要尽量保留：

- 路由原因
- 事件轨迹
- 失败重试记录

### 2）必须考虑幂等和重复处理

事件系统天然会碰到重复投递、重试等问题。

### 3）要设计好降级路径

当路由失败或无人处理时，系统应有 fallback 机制。

## 6.7 适合什么时候用

当你面对的是：

> “系统不是围绕一个固定任务做集中拆解，而是有很多事件不断发生，需要不同 Agent 按需接力处理。”

这时优先考虑 Message Bus。

# 7. 模式五：Shared State

## 7.1 模式定义

Shared State 模式中，没有一个强中心来调度所有协作。相反：

- 多个 Agent 直接读写同一个共享存储
- 这个共享存储可以是数据库、文档系统、文件系统、知识库等
- Agent 会根据共享状态中的最新信息继续工作

这种模式也常被看作“blackboard”式协作。

## 7.2 它解决什么问题

适合：

- 多个 Agent 需要实时利用彼此的新发现
- 信息不适合都绕经一个中心节点
- 工作像是共同建设一个不断更新的知识面

它解决的是：

> 协作并不总是线性流动的，有时更像多方围绕同一张白板持续增量推进。

## 7.3 典型场景

文章举的例子是研究综合：

- 一个 Agent 看论文
- 一个看行业报告
- 一个看专利
- 一个看新闻
- 它们把发现都写入共享知识库
- 其他 Agent 再基于这份共享知识继续分析

类似场景包括：

- 竞品情报知识板
- 多 Agent 研究助手
- 联合推理与证据累积
- 长文档协作提炼

## 7.4 优点

### 1）没有中心瓶颈

Agent 可以更直接地共享中间产物。

### 2）有利于知识累积

共享状态本身就会沉淀成资产。

### 3）适合探索型、研究型任务

尤其当你一开始并不知道信息会如何流动时，shared state 比固定流程更自然。

## 7.5 局限与风险

### 1）容易重复劳动

如果没有明确协调，多个 Agent 可能在做相似的事情。

### 2）容易出现方向冲突

不同 Agent 可能往共享状态里写入相互矛盾的内容。

### 3）最危险的是反应式循环

文章特别强调这一点：

- A 写了东西，触发 B
- B 更新后又触发 A
- A 再次响应
- 如此循环

系统表面上很忙，但其实没有收敛，只是在消耗资源。

## 7.6 设计时要注意什么

### 1）终止条件必须前置设计

不能等系统跑飞后再补“什么时候停”。必须事先定义：

- 多久停止
- 达到什么状态算完成
- 什么算没有新增信息
- 什么情况下要人工介入

### 2）共享状态尽量结构化

如果所有内容都只是自由文本，后续 Agent 很难有效复用。

### 3）要考虑写入冲突和版本控制

特别是多个 Agent 同时更新同一区域时。

## 7.7 适合什么时候用

当你面对的是：

> “多个 Agent 需要持续共享发现，并围绕共同知识面逐步推进任务。”

这时优先考虑 Shared State。

# 8. 五种模式怎么比较

为了便于实际选型，可以把五种模式放在一起看。

## 8.1 如果你的重点是“提高单个输出质量”

优先考虑：

- **Generate–Verify**

因为问题不在于任务拆分，而在于结果是否足够好。

## 8.2 如果你的重点是“拆一个复杂任务”

优先考虑：

- **Orchestrator–Workers**

因为它最适合中心规划、分工执行、最后汇总。

## 8.3 如果你的重点是“长期并行、多角色持续负责”

优先考虑：

- **Agent Teams**

因为它强调长期角色和持续上下文。

## 8.4 如果你的重点是“事件持续发生、系统不断扩展”

优先考虑：

- **Message Bus**

因为它最适合事件驱动与可扩展接入。

## 8.5 如果你的重点是“共同积累知识和中间发现”

优先考虑：

- **Shared State**

因为它强调协作式知识面，而不是固定线性流程。

# 9. 文章给出的关键选型思路

文章并没有让读者背模式定义，而是给了几个非常实用的判断问题。

## 9.1 任务应该按什么拆

不要先按角色名词拆，而要先问：

- 每个部分需要什么上下文？
- 哪些上下文可以被隔离？
- 哪些信息必须共享？

这是所有 multi-agent 设计的基础。

## 9.2 Orchestrator–Workers 还是 Agent Teams？

看的是：

- worker 是短期子任务执行者
- 还是长期保留上下文的稳定成员

如果需要长期记忆和连续工作，更偏 Agent Teams。

## 9.3 Orchestrator–Workers 还是 Message Bus？

看的是：

- 流程是不是预先定义的
- 还是由外部事件动态驱动的

前者偏 Orchestrator，后者偏 Message Bus。

## 9.4 Agent Teams 还是 Shared State？

看的是：

- 各成员是否主要独立推进
- 还是需要频繁共享中间发现

后者越强，就越偏 Shared State。

## 9.5 Message Bus 还是 Shared State？

看的是：

- 你在处理的是离散事件流
- 还是在构建不断增长的公共知识库

---

# 10. 文章最重要的工程启发

我认为这篇文章最有价值的，不只是列出了五种模式，而是提醒了以下几点：

## 10.1 多 Agent 设计本质上是软件架构问题

关键问题不是“让模型更聪明”，而是：

- 上下文边界
- 信息流
- 收敛机制
- 终止机制
- 冲突控制
- 可观测性

## 10.2 越复杂的协同结构，越需要治理机制

如果只有“多个 Agent 能对话”而没有：

- 明确边界
- 明确产物
- 明确停止标准
- 明确失败处理

系统大概率会变成昂贵但混乱的自动化。

## 10.3 大多数真实系统最后会是混合架构

例如：

- 外层是 Orchestrator–Workers
- 某些步骤用 Generate–Verify 做质检
- 某些异步事件通过 Message Bus 传播
- 研究阶段用 Shared State 累积信息

所以这五种模式更像“积木”，而不是互斥选项。

## 10.4 从简单模式起步是务实策略

作者的建议非常现实：

- 不要一开始就设计最复杂方案
- 从最简单可行的模式开始
- 真遇到瓶颈，再升级结构

很多情况下，**Orchestrator–Workers 会是最佳起点**。

# 11. 一个简明对照表

| 模式 | 核心结构 | 最适合的问题 | 主要优点 | 主要风险 |
|---|---|---|---|---|
| Generate–Verify | 生成 + 验证 + 迭代 | 输出质量控制 | 简单、有效、易接入 | 标准不清、难收敛 |
| Orchestrator–Workers | 中心拆解 + 子任务执行 + 汇总 | 可明确拆分的复杂任务 | 最稳、易控、易收敛 | 主控瓶颈、细节丢失 |
| Agent Teams | 长期 worker 并行推进 | 长周期、多模块工作 | 连续性强、能积累专长 | 冲突、协调复杂、资源争用 |
| Message Bus | 发布 / 订阅 / 路由 | 事件驱动系统 | 解耦、扩展性强 | 调试难、路由易错 |
| Shared State | 围绕共享存储协作 | 研究、知识累积、协同探索 | 共享直接、无中心瓶颈 | 循环、冲突、重复劳动 |

---

# 12. 最终结论

这篇文章想传达的最核心结论可以概括成四句话：

1. 多 Agent 不是默认更好，只有当拆分真的带来收益时才值得做。
2. 架构设计时最重要的不是 Agent 数量，而是上下文切分和信息流动。
3. 不同问题结构对应不同协调模式，没有单一最优架构。
4. 工程上最稳妥的策略，是从最简单可行模式开始，必要时再组合和演进。

如果只记一句话，可以记：

> **多 Agent 系统的难点不在“多”，而在“协作如何收敛”。**
